# 简历优化规则

一份优秀简历应该做到：

1. 一页纸优先
2. 项目放在最重要位置
3. 每个项目包括：
- 项目简介
- 技术栈
- 负责内容
- 项目成果

描述时尽量使用：

设计
开发
实现
优化
部署
维护

避免：

负责...

推荐：

独立开发...
设计实现...

每一条最好体现成果。

例如：

Redis缓存热点数据，接口响应时间由600ms降低至120ms。

多使用数字：

提高30%
降低50%
支持1000并发

# 百分制评分规则

简历评分使用 100 分制，不使用 A、B、C 等模糊等级代替分数。岗位匹配度 25 分，项目与工作经历 30 分，技能可信度 15 分，成果与证据 15 分，结构与可读性 10 分，基础信息完整性 5 分。评分必须说明扣分原因，不能因为文字更长就提高分数。

岗位匹配度关注目标岗位关键词、经验层级和核心职责是否一致。项目与工作经历关注业务背景、个人职责、技术动作、难点与交付物。技能可信度要求技能在经历中有对应使用证据。成果与证据优先接受真实数据、测试结果、接口数量、交付文档或部署记录；没有事实依据时不能补造数字。

# 真实性与保守优化

AI 只能重组、精炼和突出简历中已经存在的事实。原文没有出现的公司、岗位、日期、技术、项目规模、并发量、性能提升、用户数量、收入或团队规模，必须作为待确认信息，不能直接写进最终简历。

用户确认的信息可以写入最终简历，但应保留确认记录。用户跳过或没有回答的问题不得推断。日期统一格式不属于新增事实，但不能改变起止月份。

# 项目经历验收标准

一段合格的后端项目经历至少应回答：项目解决什么问题、候选人负责什么、采用什么技术动作、如何保证数据和接口正确、最终形成什么可验证交付物。技术栈列表不能替代职责描述。

可验证表达包括：实现鉴权流程、设计表结构和唯一约束、增加数据库迁移、拆分异步 worker、编写接口测试、形成 OpenAPI 文档、完成 Docker 部署。没有测试数据时，不应改写成“性能提升 50%”。

# ATS 与关键词边界

关键词应自然出现在技能或项目上下文中。只在技能列表堆砌 FastAPI、Redis、Docker 而经历中没有任何使用说明，会降低可信度。目标 JD 未要求的技术不应为了关键词覆盖强行加入。

同义词可以规范化，例如 REST API 与 RESTful API、MySQL 数据库与 MySQL，但不得把“了解”改成“精通”，不得把课程练习改成生产项目。

# 不同经验层级

应届和初级候选人重点展示基础扎实、项目完整、测试意识和学习能力。1 至 3 年经验重点展示独立负责模块、数据一致性、异常处理、部署与排错。中高级候选人还应展示系统设计、性能治理、稳定性、跨团队协作和业务结果。

优化时必须与用户真实经验层级一致。不能用高级架构术语掩盖缺少实际证据的问题。

# 常见扣分项

常见扣分项包括：联系方式缺失、目标方向混乱、时间线冲突、项目职责空泛、技能没有证据、重复描述、编造数字、把团队成果全部写成个人成果、技术名词错误、段落过长、无关经历占比过高、错别字和格式不一致。

# 优化输出验收

最终正文必须包含用户确认采用的补全内容。修改项应能定位原文和优化后文本，优化前内容必须是完整原简历，不能用第一条修改记录代替。优化摘要应说明本次实际改变了什么；如果没有改变，应明确返回未修改，不得伪造新增内容。
