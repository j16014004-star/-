# Python 后端技术能力与考核标准

本文用于职业规划打卡答疑、阶段考核、主观题评分和补强任务生成。它提供能力标准与评分依据，不保存用户秘密，也不要求考生提交密码、Token、Cookie、API Key 或生产数据库连接信息。

## Python 基础

初级应掌握常用数据结构、函数、类、异常处理、上下文管理器、类型标注、虚拟环境和包管理。能够解释可变对象与不可变对象、浅拷贝与深拷贝、迭代器与生成器的基本区别，并能写出可测试的小型模块。

中级应理解装饰器、描述符的常见用途、异步协程、并发与并行的区别、资源释放、日志和模块边界。考核重点不是背诵语法，而是能识别共享可变状态、吞异常、阻塞事件循环和资源泄漏。

验收标准：代码有清晰输入输出；异常边界明确；不使用裸 `except` 隐藏错误；具有至少一个正常路径和一个异常路径测试。

## FastAPI 与 Pydantic

初级应掌握路由、路径参数、查询参数、请求体、响应模型、状态码、依赖注入和统一异常响应。能解释为什么请求数据必须在服务端校验。

中级应理解依赖的生命周期、认证依赖、异步与同步端点的差异、中间件适用边界、OpenAPI 契约和文件上传安全。路由层只负责协议转换，业务规则放在 service，数据库操作集中在 CRUD 或仓储层。

验收标准：越权资源返回 404 或安全的拒绝响应；响应不泄露服务器绝对路径和内部异常；Pydantic Schema 限制长度、枚举和数值范围；核心接口有自动化测试。

## SQLAlchemy、MySQL 与事务

初级应掌握表、主键、外键、唯一约束、索引、分页、基本事务和 ORM 查询。能够解释 Session 的用途以及为什么请求结束需要提交或回滚。

中级应掌握异步 Session 生命周期、行锁、幂等写入、N+1 查询、复合索引、事务隔离和 Alembic 迁移。并发创建相同资源时，应依靠数据库唯一约束或事务锁保证最终一致，而不是只做一次查询判断。

验收标准：结构变化有迁移文件；跨用户查询带 `user_id` 条件；CRUD 层不随意提交事务；批量操作避免循环查询；失败路径可回滚。

## Redis、缓存与限流

Redis 可用于缓存、验证码、短期会话、幂等键、任务状态、限流和分布式锁。使用缓存必须说明 key、过期时间、失效策略和数据一致性边界。

考核时应能区分缓存穿透、击穿和雪崩。分布式锁需要唯一持有者标识和安全释放，不能把 `SETNX` 成功当成永久所有权。敏感认证信息不应以明文长期保存。

验收标准：缓存不可用时有明确降级；设置 TTL；不会因缓存命中绕过用户权限；限流能说明作用维度和窗口。

## JWT 与权限安全

JWT 登录流程包括凭证校验、签发短期 access token、可控 refresh token、服务端资源鉴权和退出策略。JWT payload 不应放密码或敏感个人信息。

考核重点：认证只证明身份，授权决定用户能否访问资源；任何简历、规划、任务和附件查询都必须使用服务端当前用户 ID 限制；不能信任客户端传入的 user_id。

验收标准：过期、伪造和错误类型 Token 被拒绝；refresh token 可撤销；日志不打印完整 Token；跨用户资源访问有测试。

## 异步任务与 Worker

耗时 AI、文件解析、爬虫和邮件任务应从 HTTP 请求中拆出。接口先持久化任务，再启动 worker，避免 worker 使用新数据库会话时找不到尚未提交的数据。

任务状态至少包括等待、执行、成功和失败；成功保存 result_id，失败保存安全错误信息。生产环境还需要超时、重试、幂等、并发限制、心跳和死信处理。

验收标准：重复请求不会重复调用模型；worker 失败会回写任务；接口可轮询；外部调用前记录审计时间；Token 用量可追踪。

## pytest 与测试策略

测试应覆盖 Schema 校验、核心 service、数据库约束、权限隔离、幂等、状态流转和外部服务失败。AI 测试使用 Fake Gateway 验证提示词与结构化输出，不应在普通回归测试中产生真实费用。

高风险功能需要测试跨用户 404、重复提交、并发边界和失败回滚。测试数量不是目标，关键业务状态是否被覆盖才是目标。

## Docker、Linux 与部署

候选人应能说明环境变量、镜像构建、容器启动、端口、日志、健康检查和数据库迁移。密钥只通过受控环境变量或密钥服务提供，不写入镜像和 Git。

生产部署中 API、worker、MySQL、Redis 和 Qdrant 应有清晰边界。Qdrant 本地文件模式适合单机开发；多进程或多实例部署应使用独立 Qdrant 服务。

## API 设计、幂等与错误处理

资源路径使用名词，HTTP 方法表达动作语义。创建耗时任务可以返回任务 ID，客户端轮询状态。重复确认、重复保存和重复进入下一阶段应返回同一结果或安全地保持现状。

客户端输入错误通常返回 422，资源不存在或不属于当前用户返回 404，状态冲突返回 409，外部服务不可用返回 503。不能把所有异常都转换成 500，也不能把内部堆栈返回前端。

## 代码题评分 Rubric

代码题永远只做静态文本分析，后端不得执行用户提交的代码。评分维度包括正确性 40%、边界与异常 20%、安全性 15%、可维护性 15%、测试意识 10%。题目满分较小时按比例换算。

如果答案存在危险命令、硬编码秘密、SQL 注入、越权、资源未释放或吞异常，应在对应维度扣分并说明原因。代码风格不能掩盖功能错误。

## 简答题评分 Rubric

简答题评价概念准确、关键点覆盖、场景适用性和风险意识。只罗列术语但无法解释因果关系，不应获得高分。答案与参考表述不同但原理正确，可以得分。

评分理由必须指出已覆盖要点、缺失要点和改进方向。不能只返回“正确”“不完整”等无依据结论。

## 阶段考核组卷规则

每次考核生成 6 至 10 题，总分 100，覆盖客观题、简答题和代码分析题。题目应与当前阶段任务、验收标准、用户打卡、历史答疑和项目交付物对应。

客观题必须保存标准答案并使用规则评分。主观题必须保存参考答案和 Rubric，由 AI 按题目满分评分。禁止要求真实密钥、生产数据或执行代码。

通过分数为 70。未通过后的补强任务应针对具体错题和薄弱能力，完成补强后重新组卷，避免简单重复相同题目。

## 常见薄弱点与补强建议

只会 CRUD：补充事务、权限、幂等、异常处理和测试任务。异步概念薄弱：分析阻塞调用进入事件循环的影响，并完成异步数据库和 worker 实践。数据库薄弱：设计带唯一约束、索引和迁移的业务表，并解释查询路径。

安全薄弱：补充 JWT、越权、输入校验、日志脱敏和文件路径安全练习。部署薄弱：完成 Docker Compose、健康检查、迁移和日志查看。测试薄弱：为成功、失败、重复和跨用户路径编写测试。
