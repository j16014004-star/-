# 职业规划确认与打卡接口

> 本文只记录本次新增的后端接口。所有接口需要 JWT 登录态，且只能操作当前用户自己的规划与任务。

## 1. 确认采用职业规划

`POST /api/career-plans/{plan_id}/accept`

处理要求：

- 把规划标记为 `accepted`，记录确认时间。
- 根据 `learning_path.stages`、`action_plan` 和 `review_points` 生成每日/每周执行任务。
- 同一规划重复确认必须幂等，不重复创建执行计划和任务。

返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "plan_id": 501,
    "status": "accepted",
    "accepted_at": "2026-07-16T14:30:00",
    "execution_plan_id": 601
  }
}
```

## 2. 根据用户反馈重新生成

`POST /api/career-plans/{plan_id}/regenerate`

请求：

```json
{
  "feedback": "每周只有 5 小时，学习任务太多，希望先强化 FastAPI 和数据库。",
  "focus_areas": ["learning_intensity", "learning_path"]
}
```

`focus_areas` 可选值：`target_role`、`learning_intensity`、`learning_path`、`project_tasks`、`job_search`。

处理要求：把原规划、用户原始档案、知识库召回内容以及本次反馈一起交给 CareerPlanningAgent；生成新的规划记录，不覆盖原规划。返回任务后沿用现有 `GET /api/ai/tasks/{task_id}` 轮询。

返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "task_id": "task_career_regenerate_xxx",
    "status": "pending",
    "result_id": null,
    "plan_id": 502,
    "previous_plan_id": 501,
    "poll_after_seconds": 1
  }
}
```

## 3. 获取当前执行计划

`GET /api/career-plan-executions/current`

没有已确认计划时建议返回 HTTP 404，`message` 为“暂无正在执行的职业规划”。

返回：

## 4. 任务打卡/撤销/暂缓

`POST /api/career-plan-executions/tasks/{task_id}/check-in`

请求：

```json
{
  "status": "completed",
  "note": "完成了接口练习，并补充了异常处理测试。"
}
```

- `completed`：完成打卡。
- `pending`：撤销完成或恢复待办。
- `skipped`：暂缓任务。
- `note` 可选，最大 300 字。

返回完整的最新 `CareerExecutionOverview`，结构与“获取当前执行计划”的 `data` 相同，便于前端立即刷新进度、连续天数和最近记录。

## 数据与业务要求

- 一个用户同一时间只能有一个 `active` 执行计划；确认新计划时，旧计划可改为 `paused`。
- 打卡接口需要幂等，同一状态重复提交不能重复累计完成数或连续天数。
- 进度、连续打卡天数、周次由后端计算，前端只展示。
- 重新生成必须保留历史规划和用户反馈，方便追踪 Agent 调整依据。
- 所有资源均按当前 JWT 用户校验，越权访问返回 404。
