# 个人中心后端接口

> 本文只描述个人中心前端需要的接口。除登录外，全部接口必须使用 Bearer JWT，并且只能操作当前用户。统一响应格式：`{ "code": 200, "message": "success", "data": ... }`。

## 1. 获取个人资料

沿用现有接口：`GET /api/auth/userinfo`

返回 `data`：

```json
{
  "id": 1,
  "username": "张三",
  "email": "user@example.com",
  "phone": "13800138000",
  "avatar": "/uploads/avatars/1/avatar.jpg",
  "status": "active",
  "email_verified": true,
  "phone_verified": false,
  "two_factor_enabled": false,
  "created_at": "2026-07-16T10:00:00",
  "last_login_at": "2026-07-16T15:20:00"
}
```

要求：

- 增加真实 `two_factor_enabled`、邮箱/手机验证状态和最后登录时间。
- 当前用户编辑页需要识别原手机号；如后端坚持返回脱敏手机号，更新资料时必须允许前端不传 `phone` 并保留原值。

## 2. 修改个人资料

`PUT /api/auth/profile`

请求：

```json
{
  "username": "张三",
  "email": "new@example.com",
  "phone": "13800138000"
}
```

- `username`：2～50 字，唯一。
- `email`：合法邮箱，唯一。
- `phone`：可选，合法手机号，唯一；不传时保持原值，传 `null` 时清空。
- 不接受客户端提交 `id`、`status`、`avatar`、角色或验证状态。
- 邮箱或手机号发生变化时，按后端安全策略把对应验证状态重置为未验证。
- 返回更新后的完整个人资料，结构同 `GET /userinfo`。

冲突建议返回 HTTP 409，并提供可展示的 `message`。

## 3. 头像上传与删除

上传：`POST /api/auth/avatar`

- 请求类型：`multipart/form-data`，字段 `file`。
- 只允许 JPG、PNG，最大 2MB。
- 校验真实 MIME 和文件头，不能只检查扩展名。
- 服务端生成文件名，保存到当前用户受控目录；替换头像时清理旧文件。

返回：

```json
{
  "code": 200,
  "message": "success",
  "data": { "avatar_url": "/uploads/avatars/1/avatar_xxx.jpg" }
}
```

删除：`DELETE /api/auth/avatar`

返回 `data: null`。重复删除需要幂等。

## 4. 修改密码

`PUT /api/auth/password`

请求：

```json
{
  "current_password": "OldPassword123",
  "new_password": "NewPassword456"
}
```

- 校验当前密码。
- 新密码 8～64 字，不能与旧密码相同。
- 使用项目当前密码哈希方案保存。
- 成功后撤销除当前会话之外的 Refresh Token；如果后端选择撤销全部 Token，需要在响应中明确要求前端重新登录。
- 不记录明文密码。

成功返回 `data: null`；当前密码错误建议返回 HTTP 400。

## 5. 两步验证

### 初始化

`POST /api/auth/two-factor/setup`

生成临时 TOTP Secret，但此时不能直接标记为启用。

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "secret": "BASE32SECRET",
    "qr_code_data_url": "data:image/png;base64,...",
    "recovery_codes": ["ABCD-EFGH", "IJKL-MNOP"]
  }
}
```

### 验证并启用

`POST /api/auth/two-factor/enable`

```json
{ "code": "123456" }
```

验证 TOTP 后保存加密 Secret，标记启用，并返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "enabled": true,
    "recovery_codes": ["ABCD-EFGH", "IJKL-MNOP"]
  }
}
```

恢复码只能保存哈希，完整内容只展示一次。

### 关闭

`POST /api/auth/two-factor/disable`

```json
{
  "password": "CurrentPassword",
  "code": "123456"
}
```

`code` 是否必填由后端安全策略决定。返回 `{ "enabled": false }`。

登录接口后续需要在启用 2FA 的账号上增加二次验证流程，但不属于本页面接口范围。

## 6. 登录记录

`GET /api/auth/login-logs?page=1&page_size=20`

返回分页结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [
      {
        "id": 101,
        "login_time": "2026-07-16T15:20:00",
        "ip_address": "127.0.0.1",
        "location": "陕西 西安",
        "device": "Desktop",
        "browser": "Chrome",
        "operating_system": "Windows",
        "status": "success"
      }
    ],
    "total": 1,
    "page": 1,
    "page_size": 20
  }
}
```

后端需在每次登录成功或失败时写入 `login_logs`。解析 User-Agent 时失败可返回原始设备摘要，但不得返回 Token。

## 7. 通知偏好

获取：`GET /api/auth/preferences`

修改：`PUT /api/auth/preferences`

数据结构：

```json
{
  "email_notifications": true,
  "push_notifications": true,
  "ai_report_notifications": true
}
```

修改接口允许提交部分字段，返回完整最新偏好。首次查询没有记录时自动返回默认值。

## 8. 删除账户

`DELETE /api/auth/account`

请求 Body：

```json
{
  "password": "CurrentPassword",
  "confirmation": "确认删除"
}
```

要求：

- 再次验证密码和固定确认文字。
- 使用项目既定的逻辑删除或账户清理策略。
- 撤销全部 Refresh Token。
- 处理简历文件、头像、职业规划附件、AI 任务和其他用户资源，不留下可被其他用户访问的数据。
- 操作需要事务或可恢复的清理流程，失败不能返回成功。

成功返回 `data: null`，前端随后清除本地登录状态并跳转登录页。

## 9. 现有退出登录

沿用：`POST /api/auth/logout`

建议明确是只撤销当前 Refresh Token，还是退出所有设备。页面当前按“退出当前账户”展示。

## 数据库建议

- `users` 增加真实验证状态、最后登录时间和两步验证字段，TOTP Secret 必须加密保存。
- 新增 `user_preferences`，一个用户一条记录。
- 复用现有 `login_logs` 表，并在登录流程真实写入。
- 恢复码建议单独表或加密/哈希 JSON 字段，并记录是否使用。

## 验收重点

- 所有资源按 JWT 当前用户隔离。
- 头像不能以 Base64 长字符串写入 `users.avatar`，数据库只保存受控 URL/相对路径。
- 修改密码和删除账户只有后端成功后前端才提示成功。
- 两步验证初始化不等于启用，必须验证首个动态码。
- 邮箱、手机号和用户名唯一冲突返回明确错误。
