# 岗位推荐模块前端接口文档

更新时间：2026-07-16

## 1. 前端实现状态

岗位推荐页面的正式接口链路已全部接入，页面不使用岗位推荐 Mock。

- 简历下拉框同时展示“原始简历”和“优化简历”两个分组；原始简历只展示 `status=completed` 的记录，优化简历只展示已经保存的版本。
- 如果用户存在已保存的优化简历，页面默认优先选择优化简历；用户仍可切换回原始简历。
- 具体岗位支持固定选项、搜索、直接输入后回车创建，以及“新增自定义岗位”弹窗。
- 自定义岗位仍以普通字符串 `target_role` 提交，不需要岗位 ID，也不需要新增岗位字典接口。
- 目标岗位最多 100 个字符，目标城市为必填项。
- 正式流程只调用一次登录启动接口，不调用旧的 `/api/jobs/recommend/start`。
- 支持登录轮询、推荐任务轮询、历史任务恢复、分页结果、岗位详情和投递意向记录。
- 页面隐藏时暂停轮询，重新显示后自动恢复；组件销毁时清理定时器。

统一响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

所有接口均携带：`Authorization: Bearer <access_token>`。

## 2. 页面实际调用接口

### 2.1 获取可用简历

`GET /api/resumes?page=1&page_size=100`

用途：初始化“原始简历”分组。前端只保留 `status=completed` 的简历。

### 2.2 获取已保存的优化简历

`GET /api/resume-optimizations/saved?page=1&page_size=100`

用途：初始化“优化简历”分组。前端使用以下字段：

```json
{
  "items": [
    {
      "id": 26,
      "resume_id": 7,
      "title": "Python后端优化简历",
      "is_saved": true,
      "saved_at": "2026-07-16T18:30:00",
      "created_at": "2026-07-16T18:00:00"
    }
  ],
  "total": 1,
  "page": 1,
  "page_size": 100
}
```

字段含义：

- `id`：优化简历版本 ID，后续作为 `resume_optimization_id` 提交。
- `resume_id`：该优化版本所属的原始简历 ID。
- `title`：下拉框展示名称。
- `saved_at`：页面展示的保存时间；为空时使用 `created_at`。

优化简历列表接口失败时，不影响用户继续使用原始简历。

### 2.3 获取招聘平台

`GET /api/job-platforms`

前端使用字段：

```json
{
  "items": [
    {
      "source": "58",
      "name": "58同城",
      "enabled": true,
      "login_required": true,
      "login_status": "logged_in"
    }
  ]
}
```

`login_status` 支持：`not_logged_in`、`waiting_login`、`logged_in`、`expired`、`failed`、`unknown`。

### 2.4 恢复当前推荐任务

原始简历：

`GET /api/jobs/recommend/current?resume_id=7&resume_source=original&source=58`

优化简历：

`GET /api/jobs/recommend/current?resume_id=7&resume_source=optimized&resume_optimization_id=26&source=58`

用途：进入页面、刷新页面、切换简历或平台时恢复最近任务。

无任务返回：

```json
{
  "task": null
}
```

有任务时，`task` 结构与“查询推荐任务”一致。

后端恢复任务时需要同时匹配 `resume_id`、`resume_source`、`resume_optimization_id` 和 `source`，避免同一原始简历的多个优化版本相互覆盖。

### 2.5 启动平台登录和岗位推荐

`POST /api/job-platforms/login/start`

请求：

```json
{
  "source": "58",
  "resume_id": 7,
  "resume_source": "optimized",
  "resume_optimization_id": 26,
  "target_role": "大模型应用开发工程师",
  "target_city": "西安",
  "limit": 20
}
```

说明：

- `target_role` 是用户选择或自定义输入的岗位名称字符串。
- 前端不会传岗位 ID。
- `target_role`、`target_city` 在前端均为必填。
- `resume_id` 始终是原始简历 ID，保持现有外键和接口兼容。
- `resume_source` 为 `original` 或 `optimized`。
- 选择原始简历时，传 `resume_source=original`，不传 `resume_optimization_id`。
- 选择优化简历时，传 `resume_source=optimized`，并传对应的 `resume_optimization_id`。后端必须使用该优化版本的最终保存内容进行技能提取、关键词生成和岗位匹配，不能继续读取原始简历正文。
- 后端应在首次响应中返回 `recommend_task_id`。
- `login_mode=server_browser` 时，前端只展示登录提示，不打开 `login_url`。

前端使用的返回字段：

```json
{
  "login_session_id": "session_xxx",
  "source": "58",
  "source_name": "58同城",
  "status": "waiting_login",
  "login_mode": "server_browser",
  "login_url": "https://passport.58.com/login",
  "expires_at": "2026-07-16T18:10:00",
  "error_message": null,
  "recommend_task_id": "task_xxx",
  "recommend_status": "pending",
  "recommend_poll_after_seconds": 2,
  "poll_after_seconds": 2
}
```

### 2.6 查询登录状态

`GET /api/job-platforms/login/{loginSessionId}`

返回结构与启动接口一致。前端处理：

- `waiting_login`：继续轮询。
- `logged_in`：停止登录轮询，使用 `recommend_task_id` 查询任务。
- `expired`：停止轮询并提示重新登录。
- `failed`：停止轮询并展示 `error_message`。

### 2.7 查询推荐任务

`GET /api/jobs/recommend/tasks/{taskId}`

前端需要字段：

```json
{
  "task_id": "task_xxx",
  "status": "matching",
  "progress": 75,
  "source": "58",
  "source_name": "58同城",
  "resume_id": 7,
  "resume_source": "optimized",
  "resume_optimization_id": 26,
  "target_role": "大模型应用开发工程师",
  "target_city": "西安",
  "extracted_skills": ["Python", "FastAPI"],
  "search_keywords": ["大模型应用开发", "AI应用开发"],
  "total_found": 20,
  "total_saved": 18,
  "total_matched": 9,
  "failure_code": null,
  "error_message": null,
  "crawl_diagnostics": {
    "query_count": 2,
    "raw_items": 20,
    "parsed_items": 18,
    "accepted_items": 9,
    "invalid_items": 2,
    "duplicate_items": 0,
    "no_result_queries": 0
  }
}
```

任务状态支持：`pending`、`crawling`、`matching`、`success`、`no_results`、`need_login`、`failed`。

失败码已映射：`no_exact_results`、`no_matching_jobs`、`parse_failed`、`crawl_failed`。

### 2.8 获取推荐结果

`GET /api/jobs/recommend/tasks/{taskId}/results?page={page}&page_size={pageSize}`

用途：任务成功或无结果后加载岗位。`no_results` 应返回 HTTP 200 和空数组。

```json
{
  "task_id": "task_xxx",
  "resume_id": 7,
  "resume_source": "optimized",
  "resume_optimization_id": 26,
  "source": "58",
  "source_name": "58同城",
  "target_role": "大模型应用开发工程师",
  "target_city": "西安",
  "extracted_skills": ["Python", "FastAPI"],
  "search_keywords": ["AI应用开发"],
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 9
}
```

岗位项至少需要：`id`、`company`、`title`、`salary_min`、`salary_max`、`city`、`experience_required`、`education_required`、`skills`、`match_score`、`match_reasons`、`source`、`source_name`、`source_url`、`crawl_time`。

### 2.9 获取岗位详情

`GET /api/jobs/{jobId}`

用途：点击岗位卡片后打开详情抽屉。返回单个完整岗位对象，字段与结果列表岗位项一致，并建议提供完整 `description`。

### 2.10 记录投递意向

`POST /api/jobs/{jobId}/apply`

请求：

```json
{
  "resume_id": 7,
  "resume_source": "optimized",
  "resume_optimization_id": 26
}
```

用途：只在本系统记录用户投递意向，不要求后端自动操作招聘平台。后端应保存本次实际使用的简历来源和优化版本，便于后续查看投递记录时确认使用的是哪一版简历。

建议返回：

```json
{
  "id": 101,
  "job_id": 1,
  "resume_id": 7,
  "resume_source": "optimized",
  "resume_optimization_id": 26,
  "status": "pending",
  "apply_type": "manual",
  "applied_at": "2026-07-16T20:00:00",
  "agent_task_id": null
}
```

## 3. 兼容接口说明

- `POST /api/jobs/recommend/start`：前端仍保留旧类型和封装供旧代码兼容，但当前岗位推荐页面不会调用。
- `GET /api/jobs/recommendations`：由首页和简历优化页读取岗位标题使用，不属于当前岗位推荐任务链路，不能替代任务结果接口。

## 4. 前端接口检查结论

当前岗位推荐页面所需接口均已有前端封装并已接入，没有发现仍以本地假数据代替、按钮未调用接口或任务状态未处理的功能。后端只需按本文字段实现并保持统一响应结构即可对接。

本次新增的后端对接重点是：岗位推荐启动、当前任务恢复、任务结果和投递记录需要识别 `resume_source` 与 `resume_optimization_id`。如果选择优化简历，后端必须读取保存后的 `optimized_content`，而不是原始简历解析文本。
